# 第三方大模型对接接口说明

## 一、基础说明

### 1.1 系统说明

机器人系统分为 **Robot 系统**和 **Agent 系统**，两个系统接入第三方大模型的方式有所不同。

#### Robot 系统

Robot 系统接入其他大模型时，只能使用本文所述的**第三方大模型接口**。第三方服务需要按照接口约定接收 Robot 系统发起的请求，并将大模型生成的结果返回给 Robot 系统。

#### Agent 系统

Agent 系统支持以下两种接口形式：

- 第三方大模型接口；
- OpenAI 兼容类型接口。

需要特别说明的是，Agent 系统本身通过大模型提供 MCP 服务，并由系统大模型进行问题理解、任务规划和能力调用。因此，Agent 系统收到用户问题后，会先由系统内部进行判断和处理，**并非所有问题都会请求所配置的第三方大模型接口或 OpenAI 兼容接口**。

只有当 Agent 系统根据问题内容和任务规划，判断需要调用第三方大模型能力时，才会向对应接口发起请求。

### 1.2 统一流式响应要求

第三方大模型接口和 OpenAI 兼容接口均**只支持流式响应**，不支持非流式响应。

所有接口必须满足以下时延要求：

- 从请求成功建立到返回首个有效数据帧，耗时不得超过 **5 秒**；
- 流式输出过程中，任意相邻两个数据帧之间的间隔不得超过 **5 秒**；
- 如果模型暂时没有生成新的文本，接口服务仍应在 5 秒内返回符合对应协议的数据帧，避免连接因长时间无数据而超时；
- 模型生成结束后，应立即发送结束帧并关闭本次流式响应。

> 以上要求同时适用于 Robot 系统使用的第三方大模型接口，以及 Agent 系统使用的第三方大模型接口和 OpenAI 兼容接口。

## 二、OpenAI 兼容接口

### 2.1 适用范围

OpenAI 兼容接口按照 OpenAI Chat Completions 协议发送请求和解析响应，可用于接入 OpenAI、DeepSeek，以及其他兼容该协议的大模型服务。

本接口主要供 Agent 系统使用。Agent 系统会将配置的大模型服务注册为一项 MCP 能力，再由系统大模型根据 MCP 的名称和职责描述判断是否调用。因此，完成接口配置不代表所有用户问题都会发送给该模型。

### 2.2 配置项

接入一个 OpenAI 兼容模型时，需要提供以下配置：

| 英文字段 | 中文名称 | 必填 | 说明 |
| --- | --- | --- | --- |
| `name_en` | 英文名称 | 是 | MCP 能力的唯一英文标识。建议使用小写英文字母、数字和下划线，例如 `general_llm` |
| `name_zh` | 中文名称 | 是 | MCP 能力的中文展示名称，例如“通用大模型问答” |
| `description` | MCP 职责描述 | 是 | 描述该模型能够处理的问题、适用场景和能力边界，Agent 系统将依据此描述判断是否调用 |
| `url` | 接口地址 | 是 | 完整的 Chat Completions 请求地址，必须是可通过 HTTP 或 HTTPS 访问的 URL |
| `api_key` | API 密钥 | 是 | 模型服务商提供的访问密钥，请求时通过 Bearer Token 方式传递 |
| `model` | 模型名称 | 是 | 模型服务商支持的模型标识，具体值以服务商文档或控制台为准 |
| `system_prompt` | 系统级提示词 | 否 | 调用模型时附加的系统提示词，用于限定模型身份、回答风格或业务范围；未填写时不注入额外的 `system` 消息 |

> `api_key` 属于敏感信息，配置界面和日志中应进行脱敏处理，不应以明文输出。

### 2.3 MCP 名称和职责说明

`name_en`、`name_zh` 和 `description` 用于定义该大模型在 Agent 系统中的 MCP 能力。它们不作为 OpenAI Chat Completions 的标准请求字段直接发送给第三方模型。

- **英文名称**：作为 MCP 能力的唯一标识，应简短、稳定，并能体现主要用途。
- **中文名称**：用于管理界面展示，方便配置人员识别能力。
- **MCP 职责描述**：用于告诉 Agent 系统“该能力可以做什么、应在什么情况下调用、哪些事情不负责”。描述越清晰，系统越容易做出正确的调用判断。

推荐的职责描述结构：

```text
负责处理：[能力范围]。
适用于：[调用场景]。
输入内容：[主要输入]。
输出内容：[预期输出]。
不负责：[能力边界或禁止事项]。
```

配置示例：

```json
{
  "name_en": "general_llm",
  "name_zh": "通用大模型问答",
  "description": "负责处理通用知识问答、文本总结和内容生成。适用于无需调用机器人硬件能力的普通对话。不负责机器人导航、运动控制、设备设置和其他需要执行实体动作的任务。",
  "url": "https://api.example.com/v1/chat/completions",
  "api_key": "YOUR_API_KEY",
  "model": "YOUR_MODEL_NAME",
  "system_prompt": "请使用简洁、准确的中文回答用户问题。"
}
```

### 2.4 请求规范

#### 请求方法

```http
POST {url}
Content-Type: application/json
Authorization: Bearer {api_key}
```

其中，`{url}`、`{api_key}` 和请求体中的 `model` 分别使用配置项中的对应值。

#### 请求体

请求体按照 OpenAI Chat Completions 格式发送：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `model` | string | 是 | 配置的模型名称 |
| `messages` | array | 是 | 对话消息列表，按照上下文顺序排列 |
| `stream` | boolean | 是 | 固定为 `true`，接口只支持 SSE 流式响应 |
| `temperature` | number | 否 | 控制输出随机性，是否支持及取值范围以模型服务商为准 |
| `max_tokens` | integer | 否 | 限制最大输出 Token 数，是否支持以模型服务商为准 |

`messages` 中每条消息的基本结构如下：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `role` | string | 消息角色，常用值包括 `system`、`user`、`assistant` |
| `content` | string | 消息文本内容 |

当配置了 `system_prompt` 时，Agent 系统应将其作为 `messages` 中的首条 `system` 消息发送；未配置或内容为空时不添加该消息。

请求示例：

```json
{
  "model": "YOUR_MODEL_NAME",
  "messages": [
    {
      "role": "system",
      "content": "请使用简洁、准确的中文回答用户问题。"
    },
    {
      "role": "user",
      "content": "请介绍一下这台机器人的主要能力。"
    }
  ],
  "stream": true
}
```

### 2.5 流式响应与解析

接口仅接受 `stream` 为 `true` 的请求，响应的 `Content-Type` 应为 `text/event-stream`。服务端通过 SSE 持续返回 OpenAI 风格的数据分片：

```text
data: {"id":"chatcmpl-example","object":"chat.completion.chunk","created":1750000000,"model":"YOUR_MODEL_NAME","choices":[{"index":0,"delta":{"role":"assistant","content":""},"finish_reason":null}]}

data: {"id":"chatcmpl-example","object":"chat.completion.chunk","created":1750000000,"model":"YOUR_MODEL_NAME","choices":[{"index":0,"delta":{"content":"机器人支持"},"finish_reason":null}]}

data: {"id":"chatcmpl-example","object":"chat.completion.chunk","created":1750000000,"model":"YOUR_MODEL_NAME","choices":[{"index":0,"delta":{},"finish_reason":"stop"}]}

data: [DONE]
```

Agent 系统应逐行读取以 `data:` 开头的事件，并按以下规则解析：

1. 去除行首的 `data:` 和两侧空白字符；
2. 当内容为 `[DONE]` 时结束本次流式解析；
3. 其他内容按 JSON 解析；
4. 读取 `choices[0].delta.content` 并按接收顺序拼接；
5. 当 `finish_reason` 不为空时，认为模型生成已经结束；
6. `delta.content` 为空或字段不存在时，不应将其拼接为文本。

#### 流式帧时延要求

- 首个 SSE 数据帧必须在请求建立后的 **5 秒内**返回；
- 后续任意相邻两个 SSE 数据帧的间隔不得超过 **5 秒**；
- 无新增文本时，可以返回 `delta.content` 为空的合法数据帧以维持流式连接；
- 最后一个内容帧生成后，应及时返回带有 `finish_reason` 的结束帧，并发送 `data: [DONE]`。

### 2.6 错误响应

请求失败时，应使用与错误类型相符的 HTTP 状态码。建议响应体采用以下结构：

```json
{
  "error": {
    "message": "Invalid API key",
    "type": "authentication_error",
    "param": null,
    "code": "invalid_api_key"
  }
}
```

Agent 系统应首先判断 HTTP 状态码。HTTP 状态码不是 `2xx` 时，应停止正常响应解析，记录脱敏后的错误信息，并向调用方返回统一的模型服务异常提示。

### 2.7 服务商配置示例

#### OpenAI

```text
url: https://api.openai.com/v1/chat/completions
api_key: OpenAI 平台生成的 API Key
model: 以 OpenAI 控制台及官方模型文档为准
```

#### DeepSeek

```text
url: https://api.deepseek.com/chat/completions
api_key: DeepSeek 平台生成的 API Key
model: 以 DeepSeek 控制台及官方模型文档为准
```

不同服务商对可选参数、模型名称、上下文长度及扩展字段的支持可能不同。为保证兼容性，基础接入只应依赖 `model`、`messages`、`stream` 和标准 `choices` 响应结构。

## 三、第三方大模型接口

### 3.1 开发说明

第三方大模型接口用于将 Robot 系统或 Agent 系统的对话请求转发到第三方提供的大模型服务。第三方需要自行开发一个 HTTP 服务，接收系统发送的对话参数，调用实际的大模型，并按照本章约定返回 SSE 流式响应。

本章只定义第三方大模型对话接口所需的请求参数、SSE 响应格式、分帧规则和异常处理。其他具体业务功能不属于第三方大模型接口的必要组成部分。

第三方服务的基本处理流程如下：

```text
Robot/Agent 系统发起请求
    → 第三方服务校验请求参数
    → 第三方服务立即返回首帧
    → 第三方服务调用实际大模型
    → 将模型生成内容转换为 SSE 数据帧
    → 持续返回内容帧
    → 返回 finish_reason=stop 的结束帧
    → 关闭流式连接
```

### 3.2 接口地址

第三方服务需要提供一个可由 Robot 系统或 Agent 系统访问的 HTTP 或 HTTPS 地址。

示例服务使用以下接口：

```http
POST /v1/chat/completions
Content-Type: application/json
```

实际部署时，需要向系统提供包含协议、域名、端口和路径的完整 URL，例如：

```text
https://llm.example.com/v1/chat/completions
```

#### 自定义请求头

第三方大模型接口支持配置自定义 HTTP 请求头。系统调用第三方服务时，会将已配置的请求头添加到请求中。

请求头可以用于身份认证、租户识别、调用来源识别或其他第三方服务要求的访问限制。请求头的名称和值由第三方服务提供方定义。

如需使用标准 Bearer Token 进行访问控制，可以配置 `Authorization` 请求头，其内容必须以 `Bearer` 开头：

```http
Authorization: Bearer YOUR_ACCESS_TOKEN
```

完整请求示例：

```http
POST /v1/chat/completions HTTP/1.1
Host: llm.example.com
Content-Type: application/json
Authorization: Bearer YOUR_ACCESS_TOKEN
X-Tenant-Id: YOUR_TENANT_ID
```

自定义请求头配置示例：

```json
{
  "headers": {
    "Authorization": "Bearer YOUR_ACCESS_TOKEN",
    "X-Tenant-Id": "YOUR_TENANT_ID"
  }
}
```

其中：

- `headers` 为请求头键值对象；
- `Authorization` 为可选字段，配置后第三方服务可以据此限制未授权访问；
- `Authorization` 使用 Bearer 认证时，格式固定为 `Bearer <token>`；
- 除 `Authorization` 外，还可以按业务需要配置其他请求头；
- 认证凭证属于敏感信息，配置界面、运行日志和异常信息中必须进行脱敏处理。

### 3.3 请求参数

请求体为 JSON 对象：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | string | 是 | 本次请求的唯一标识，不允许为空。第三方服务必须在所有 SSE 返回帧中原样返回该值 |
| `messages` | array | 是 | 非空的对话消息数组，按照对话发生顺序排列 |
| `model` | string | 否 | 请求使用的模型名称。未提供时，第三方服务可以使用服务端配置的默认模型 |
| `stream` | boolean | 是 | 固定为 `true`，第三方接口只支持流式响应 |
| `max_tokens` | integer | 否 | 最大输出 Token 数。第三方服务可以将该参数透传给实际模型 |
| `temperature` | number | 否 | 模型生成随机性。第三方服务可以将该参数透传给实际模型 |

`messages` 数组中的消息对象：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `role` | string | 是 | 消息角色，常用值为 `system`、`user`、`assistant` |
| `content` | string | 是 | 消息文本内容 |

如果系统配置了系统级提示词，该提示词会作为 `role=system` 的消息放入 `messages`。第三方服务应按消息原有顺序将有效的 `role` 和 `content` 转发给实际模型。

请求示例：

```json
{
  "id": "req_8f31e8d5",
  "model": "third-party-model",
  "messages": [
    {
      "role": "system",
      "content": "请使用简洁、准确的中文回答用户问题。"
    },
    {
      "role": "user",
      "content": "请介绍一下机器人的主要功能。"
    }
  ],
  "stream": true,
  "max_tokens": 1024,
  "temperature": 0.7
}
```

### 3.4 SSE 响应格式

第三方服务必须返回 SSE 流式响应：

```http
HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache, no-transform
Connection: keep-alive
X-Accel-Buffering: no
```

每个 SSE 数据帧由 `id` 行、`data` 行和一个空行组成：

```text
id: {request_id}_{sequence}
data: {JSON 数据}

```

字段说明：

| SSE 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | string | SSE 事件标识，由请求 `id`、下划线和从 1 开始递增的帧序号组成 |
| `data` | JSON string | 本帧的大模型响应对象 |

`data` 对应的 JSON 对象结构：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | string | 原样返回请求中的 `id` |
| `created` | integer | 响应创建时间，Unix 秒级时间戳 |
| `model` | string | 本次实际使用的模型名称 |
| `choices` | array | 模型响应选项数组，当前只使用第一个元素 |

`choices[0]` 对象结构：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `index` | integer | 固定为 `0` |
| `delta` | object | 当前帧新增的内容 |
| `finish_reason` | string/null | 中间帧为 `null`，正常结束帧为 `stop` |

`delta` 对象可包含以下字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `role` | string | 仅首帧使用，值为 `assistant` |
| `content` | string | 当前帧新增的文本内容。中间帧只返回本帧新增部分，不返回已生成的完整文本 |

### 3.5 分帧规则

#### 首帧

第三方服务收到并校验请求后，应立即返回首帧，不应等待上游大模型产生文本。首帧用于确认流式连接已经成功建立：

```text
id: req_8f31e8d5_1
data: {"id":"req_8f31e8d5","created":1750000000,"model":"third-party-model","choices":[{"index":0,"delta":{"role":"assistant","content":""},"finish_reason":null}]}

```

#### 内容帧

模型产生文本后，通过一个或多个内容帧按顺序返回。内容帧的 `delta` 中只携带新增的 `content`：

```text
id: req_8f31e8d5_2
data: {"id":"req_8f31e8d5","created":1750000000,"model":"third-party-model","choices":[{"index":0,"delta":{"content":"机器人支持"},"finish_reason":null}]}

id: req_8f31e8d5_3
data: {"id":"req_8f31e8d5","created":1750000000,"model":"third-party-model","choices":[{"index":0,"delta":{"content":"语音交互和任务执行。"},"finish_reason":null}]}

```

#### 结束帧

模型生成完成后，必须返回空 `delta` 且 `finish_reason` 为 `stop` 的结束帧：

```text
id: req_8f31e8d5_4
data: {"id":"req_8f31e8d5","created":1750000000,"model":"third-party-model","choices":[{"index":0,"delta":{},"finish_reason":"stop"}]}

```

第三方大模型接口以 `finish_reason=stop` 和流式连接关闭作为响应结束标志，**不要求额外返回 `data: [DONE]`**。

### 3.6 流式时延要求

第三方大模型接口必须满足以下要求：

- 首帧必须在收到请求后的 **5 秒内**返回；
- 任意相邻两个 SSE 数据帧之间的间隔不得超过 **5 秒**；
- 如果实际模型暂时没有生成文本，应在 5 秒内返回 `delta.content` 为空的合法数据帧；
- 每一帧都必须是格式完整的 SSE 事件，不能只返回未完成的 JSON 片段；
- 模型生成完成后，应立即返回结束帧并关闭连接。

为降低首帧耗时，推荐由第三方服务立即构造首帧，再调用实际的大模型接口。

### 3.7 系统解析规则

Robot 系统和 Agent 系统按照以下规则解析第三方服务返回的数据：

1. 按 SSE 事件读取响应；
2. 读取并记录 SSE 的 `id`；
3. 将 `data` 内容解析为 JSON 对象；
4. 校验返回对象中的 `id` 是否与请求 `id` 一致；
5. 读取 `choices[0].delta.content`；
6. 按接收顺序拼接所有非空 `content`；
7. 收到 `choices[0].finish_reason=stop` 后结束解析；
8. 结束帧后关闭连接，或者等待第三方服务主动关闭连接。

系统只将 `delta.content` 作为最终回答文本进行处理。工具调用、模型内部思考内容及其他非文本扩展字段不属于本接口的基础解析范围。

### 3.8 参数校验和异常处理

第三方服务应在开始 SSE 响应前完成基础参数校验：

- 请求体不是合法 JSON：返回 HTTP `400`；
- 请求体不是 JSON 对象：返回 HTTP `400`；
- `id` 缺失或为空：返回 HTTP `422`；
- `messages` 不是非空数组：返回 HTTP `422`；
- `messages` 中没有包含有效 `role` 和 `content` 的消息：返回 HTTP `422`；
- 第三方服务缺少必要的模型配置：返回 HTTP `500`。

建议错误响应使用 JSON：

```json
{
  "detail": "请求体必须提供非空 id"
}
```

如果异常发生在 SSE 流式响应开始之后，HTTP 状态码已经无法修改。此时第三方服务应返回一个可识别的错误内容帧，再返回 `finish_reason=stop` 的结束帧并关闭连接，不能让连接一直处于等待状态。

### 3.9 第三方服务最小实现要求

一个可接入的第三方大模型服务至少需要完成以下功能：

1. 提供可访问的 `POST` 接口；
2. 接收并校验 `id`、`messages`、`model`、`stream` 等参数；
3. 在 5 秒内返回首个 SSE 数据帧；
4. 将 `messages` 转发给实际使用的大模型；
5. 从实际模型的流式响应中提取新增文本；
6. 按本文定义的 SSE JSON 结构返回文本分片；
